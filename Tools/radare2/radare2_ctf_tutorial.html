Lesson 01 · Foundations
Opening Binaries
Radare2's command language looks intimidating at first — everything is abbreviated. Once you see the pattern, it clicks fast. This lesson covers launching, modes, and the fundamental command structure.
r2 launchwrite moderadare2 -Aquit
⚙ Installation
shell
# Option A: package manager (older version, fine for learning) sudo apt install radare2 # Option B: build from source (always latest) git clone https://github.com/radareorg/radare2 cd radare2 && sys/install.sh # Verify r2 -v
🚀 Launching r2
shell
r2 ./binary # Open for analysis (read-only by default) r2 -A ./binary # Open AND auto-analyze (like 'aaa' on entry) r2 -w ./binary # Open in WRITE mode (needed for patching) r2 -d ./binary # Open in DEBUG mode (like gdb) r2 -d -A ./binary # Debug + auto-analyze r2 - # Open without a file (for scripting) # With arguments (for debug mode): r2 -d ./binary arg1 arg2
🧠 Command Structure — The Core Pattern
💡 How r2 Commands Work
Every command follows: verb + noun + modifier. Example: pdf = print disassembly of function. Append ? to anything for help: p? shows all print commands. pd? narrows it. This is the key to r2's entire CLI.
r2 command anatomy
# p = print family # pd = print disassembly # pdf = print disassembly of function # pdfs = print disassembly of function (summary) # a = analysis family # aa = analyze all # aaa = analyze all (deeper) # af = analyze function # afi = analyze function info # s = seek (move current position) # s main → seek to main # s 0x401000 → seek to address

Lesson 02 · Foundations
Navigation & Flags
In r2, your "cursor" is the current seek position. Everything you do operates at the seek position. Flags are r2's bookmarks — named labels on addresses. Understanding both is fundamental.
s (seek)flagsf commandafl
🧭 Seeking (Moving the Cursor)
r2
s # Show current position (address) s main # Seek to symbol 'main' s 0x401000 # Seek to absolute address s+10 # Seek forward 10 bytes s-10 # Seek back 10 bytes s$$ # Seek to current position (no-op, for scripts) s- # Undo seek (go back) s+ # Redo seek (go forward) s sym.win # Seek to a flag named 'win'
🚩 Flags (r2's Bookmark System)
What are Flags?
After analysis, r2 creates flags for every symbol, string, function, import, export. sym.main, str.hello_world, fcn.00401000, imp.puts. You can also create your own. They're the primary way to refer to addresses by name.
r2
f # List all flags f~main # Filter flags containing "main" (~ = grep) f~str. # All string flags f~sym. # All symbol flags f~imp. # All import flags f myflag @ 0x401234 # Create a flag at address f- myflag # Delete a flag fl # List flags with lengths
📋 Listing Functions, Symbols, Imports
r2 — after running aa or aaa
afl # list ALL analyzed functions (addr, size, name) afl~main # grep for "main" in function list afl | sort -k3 # pipe to shell sort afl~[0] # just addresses (column 0) is # list symbols (from symbol table) is~FUNC # filter to functions only ii # list imports (libc functions used) ie # entry point(s) iz # strings in data section izz # strings across ENTIRE binary
⚡ Grep Everything with ~
The ~ operator is r2's built-in grep. afl~win finds any function with "win" in the name. iz~flag finds any string containing "flag". Add ~[0] to extract column 0, ~[1] for column 1. Combine: afl~win[0] = just the address of win.

Lesson 03 · Foundations
Analysis (aa / aaa)
r2 doesn't analyze a binary automatically — you ask for it. The analysis depth is configurable. Too little and you miss functions; too much and it's slow. Here's how to get it right.
aa / aaaafxrefscall graph
🔬 Analysis Commands
r2
aa # Analyze all — quick, finds obvious functions aaa # Analyze all + extra passes (recommended for CTF) aaaa # Maximum analysis — slow but thorough af # Analyze function at current seek position af @ 0x401000 # Analyze function at specific address afr # Analyze function recursively (follows calls) afl # List all known functions (after analysis) afi # Info about current function afn new_name # Rename current function afn new_name @ 0x401000 # Rename function at address
⚠ Run aaa Before Anything
If you open a binary and pdf shows nothing or garbage, it's because you haven't analyzed yet. Always run aaa (or use r2 -A) before exploring. Stripped binaries especially need it to find function boundaries.
🔗 Cross-References (xrefs)
r2 — xrefs are CTF gold
axt # xrefs TO current address (who calls/uses me?) axt @ sym.win # Who calls win()? axf # xrefs FROM current address (what do I call?) axf @ sym.main # What does main() call? axt @ str.password # Which function references "password" string?
📊 Binary Info & Security Flags
r2 — recon commands
i # General binary info (arch, bits, OS, type) iI # Security: canary, nx, pic, relocs, stripped iS # Sections (.text, .data, .bss with perms) iS~rwx # Find executable+writable sections ii # Imports (libc calls) ie # Entry point ir # Relocations (GOT entries) iz # Strings in data section

Lesson 04 · Foundations
Disassembly — pdf, pd, pdc
The p (print) family is your most-used command set. pdf is the workhorse: disassemble an entire function with arrows, xrefs, and comments. Here's everything you need to read code in r2.
pdfpd NpdcVV graph
🔎 Disassembly Commands
r2
pdf # Disassemble entire current function pdf @ main # Disassemble main (from anywhere) pdf @ sym.win # Disassemble win() pd 20 # Disassemble 20 instructions from current pos pd 20 @ 0x401000 # 20 instructions from address pdi # Disassemble without metadata (cleaner output) pdfs # Function disassembly summary (just calls) pdc # Pseudo-C decompilation (basic but useful) pdc @ sym.main # Decompile main to C-like code
📍 Reading pdf Output
example pdf output

Lesson 05 · Memory
Reading Memory — px, pf, p commands
The px family dumps raw memory in hex. pf lets you define structs and read formatted data. Essential for understanding data layouts and finding hidden values.
px / pxw / pxqp8pfpv
💾 px — Hex Dump Family
r2
px # Hex dump at current position (default 16 bytes) px 64 # Hex dump 64 bytes px 64 @ 0x401000 # 64 bytes at address pxw 64 # Hex dump as 4-byte words pxq 64 # Hex dump as 8-byte qwords (64-bit) pxa 64 # Annotated hex dump (resolves pointers) p8 16 # Raw bytes as hex string (no formatting) pxr 64 # Dump with pointer references resolved (like telescope)
📖 Print Strings & Values
r2
ps # Print string at current position ps @ 0x402010 # Print string at address psz # Print null-terminated string pv # Print value (pointer size int) pv4 # Print 4-byte int pv8 # Print 8-byte int ? 0x41414141 # Evaluate/convert: hex→dec, dec→hex etc. ? 0x41 # Shows char, decimal, hex, binary all at once
💡 The ? Command
The ? expression evaluator is incredibly handy: ? 0x401000 + 0x50 computes offsets. ? 'A' shows all representations of 'A'. ? rip shows current rip in debug mode. Use it constantly for quick arithmetic.
🏗 pf — Formatted Print (Structs)
r2
pf # Show format specifiers help pf i # Print 4-byte int at cursor pf q # Print 8-byte qword pf s # Print string pointer pf iis # Print struct: int, int, string ptr pf 4i # Print array of 4 ints pf b # Print as binary # Example: reading a struct {int id; int score; char* name;} pf iis @ 0x601040

Lesson 06 · Memory
Strings & Searching Memory
Finding strings, patterns, and specific byte sequences. In CTFs this is how you locate flags stored in memory, find hardcoded keys, and track down interesting data without reading every byte manually.
iz / izz/ search/x bytes/R gadgets
📝 String Enumeration
r2
iz # Strings in data section (.rodata/.data) izz # Strings in ENTIRE binary (slower but thorough) iz~flag # Filter strings containing "flag" iz~CTF{ # Find flag format iz~[1] # Print only addresses (column 1) iz~[2] # Print only the strings (column 2)
🔍 Memory Search (/ command)
r2
/ hello # Search for string "hello" in memory /x 90 # Search for byte 0x90 (NOP) /x 90909090 # Search for NOP sled /x 41424344 # Search for "ABCD" as bytes /w 0x1337 # Search for 4-byte value 0x1337 /q 0x4141414141414141 # Search for 8-byte value /i hello # Case-insensitive string search /r 0x401000 # Search for references to address 0x401000 # Set search range first (faster, less noise): e search.in=dbg.maps # Search in all debug maps e search.in=section.text # Search only in .text
🧩 ROP Gadget Search
r2 — ROP with /R
/R pop rdi # Find "pop rdi; ret" gadgets /R pop rdi; ret # More specific search /R ret # All ret gadgets (for alignment) /R jmp rsp # Find jmp rsp (for stack pivots) /Rj pop rdi # Output as JSON # Alternative: use the ROP chain builder rop # Show available ROP gadgets (after analysis) rop~pop rdi # Filter ROP gadgets
💡 ROP Workflow in r2
/R pop rdi searches for the byte sequence of "pop rdi; ret" across the entire binary. Much faster than ROPgadget for quick lookups. The result shows the address — use that directly in your exploit.

Lesson 07 · Debugging
Debugging — ood, dc, dr, db
r2 has a built-in debugger. Not as polished as GDB but powerful — especially combined with analysis. Same workflow: break, step, inspect registers and memory.
r2 -ddb breakpointsdc / dso / dsidr registers
🐛 Starting Debug Mode
shell + r2
# Launch in debug mode: r2 -d ./binary r2 -d -A ./binary # Debug + analyze # Already in r2? Reopen in debug: ood # Reopen in debug mode (with debug backend) ood arg1 arg2 # Reopen with arguments oodf file # Reopen with different binary # Analyze after opening in debug: dcu main # Debug Continue Until main (faster than aaa + b main + dc) aaa # Now analyze (symbols are resolved)
⏸ Breakpoints
r2 debug
db sym.main # Breakpoint at main db 0x401166 # Breakpoint at address db sym.imp.strcmp # Breakpoint at strcmp db- # Remove breakpoint at current position db- 0x401166 # Remove specific breakpoint dbl # List all breakpoints dbe 0x401166 # Enable breakpoint dbd 0x401166 # Disable breakpoint
▶ Execution Control
r2 debug
dc # Continue (until breakpoint or end) dcu 0x401166 # Continue Until address (no bp needed) dcu sym.win # Continue until win() ds # Step into (one instruction) dso # Step over (skip calls) dss # Step skip (skip current instruction) dcr # Continue until ret (finish function) dk 9 # Send signal 9 (SIGKILL) to process
💡 dcu is your best friend
dcu main runs straight to main, saving you from single-stepping through all the libc startup code. dcu sym.imp.strcmp runs to the next strcmp call. Fastest way to reach an interesting point.
📊 Registers & Memory in Debug
r2 debug

Lesson 08 · Power Features
Patching Binaries
r2's write mode lets you modify the binary on disk — NOP out a check, change a jump direction, patch a comparison value. This is faster than scripting an exploit for simple CTF challenges.
r2 -wwx / wa / wvwao noppatch in-place
✏️ Write Commands
r2 — open with r2 -w ./binary
wx 90 # Write 0x90 (NOP) at current position wx 9090909090 # Write 5 NOPs wx 90 @ 0x401166 # Write NOP at specific address wv 0x1337 @ 0x601040 # Write 4-byte value at address wv8 0xdeadbeef @ addr # Write 8-byte value w hello\x00 @ addr # Write string with null terminator wz hello @ addr # Write null-terminated string # Write assembly instruction directly: wa nop @ 0x401166 # Assemble "nop" and write it wa jmp 0x401180 @ 0x401170 # Patch a jump wa xor eax,eax @ addr # Write xor eax,eax (set to 0)
🔫 wao — One-Instruction Patchers
r2 — convenience commands for common patches
wao nop # NOP the current instruction (handles variable sizes) wao ret # Replace instruction with ret wao ret0 # Replace with: xor eax,eax; ret (return 0) wao ret1 # Replace with: mov eax,1; ret (return 1) wao jz # Change jne/jnz to jz/je (flip jump) wao jnz # Change je/jz to jne/jnz wao trap # Replace with int3 (breakpoint)
⚡ Most Common CTF Patches
1. Bypass check: seek to the jne after a comparison, then wao nop or wao jz to flip it.
2. Skip a function: seek to the call, wao nop until the call is gone.
3. Force return value: seek to function start, wao ret1.
💾 Saving Changes
r2
# Changes in -w mode are written to disk IMMEDIATELY # Make a backup first! !cp ./binary ./binary.bak # Shell command from within r2 # To patch a copy without touching the original: !cp ./binary ./binary_patched r2 -w ./binary_patched # Open the copy

Challenge 10: Automate a XOR Scan
15 min
▼
Compile a binary with an XOR-encoded string:
// key = 42, "CTF{xor_me}" XOR 42 stored as bytes
char enc[] = {0x69,0x7e,0x6c,0x1b,0x5b,0x57,0x5b,0x4d,0x1b,0x6f,0x1d};
int main(){char b[32];scanf("%s",b);/* decode and compare */}

Using r2pipe Python:
1. Open the binary with r2pipe
2. aaa then find the enc array via iz or section scan
3. Read the bytes with p8 11 @ <addr>
4. In Python, XOR each byte with 42 to decode
5. Print the flag string
6. Bonus: use rasm2 to find the bytes of "pop rdi; ret" and verify with /x in r2 
import r2pipe
r2 = r2pipe.open("./binary")
r2.cmd("aaa")
# find addr of enc
addr = r2.cmd("afl~enc[0]").strip()
raw = r2.cmd(f"p8 11 @ {addr}")
decoded = bytes(b ^ 42 for b in bytes.fromhex(raw))
print(decoded)

Final Challenge: Full RE + Patch
capstone · 20 min
▼
Compile stripped with PIE:
gcc -s -pie -fpie -fno-stack-protector -o final final.c
with:
int v=0xdead; void win(){puts("r2_mastery");} int main(){char b[64];gets(b);if(v==0xcafe)win();}

Approach A — Pure Reverse Engineering:
1. r2 -A ./final → iI → what protections?
2. ii → what functions are imported?
3. axt @ sym.imp.puts → find win()
4. pdf @ fcn.win_addr → confirm it prints the flag
5. pdf @ main → understand the check: v must equal 0xcafe

Approach B — Patch it:
6. !cp ./final ./final_p && r2 -w -A ./final_p
7. Find the cmp instruction in main, patch to NOP or flip the jump
8. !./final_p <<< test — does it print "r2_mastery"?

Approach C — Debug:
9. r2 -d -A ./final → dcu main
10. Find v's address (dr rsp region), use wv 0xcafe @ addr_of_v
11. dc with any input → win!
iI → stripped:true, pic:true (PIE), canary:false
ii → imports: gets, puts
axt @ sym.imp.puts → shows win function address
pdf @ main → find: cmp dword [var], 0xcafe + jne

Patch: seek to jne → wao nop (twice if jne is 6 bytes)
Debug: dcu main → find v on stack → wv 0xcafe @ addr
